# 문화적합성 인터뷰 대비 — 토스증권

> STAR 기법: 상황(Situation) → 과제(Task) → 행동(Action) → 결과(Result)
> 각 핵심가치당 사례 2개 준비

---

## 1. 고객 중심

### 사례 A — 토스증권 앱 직접 관찰 → 토큰 충돌 발견
- **S**: 토스증권 [Chart] 지원을 결정하고, 실제 앱을 사용하며 차트 UI를 분석
- **T**: 지원동기에 구체적 개선 제안을 담기 위해 실사용자 관점에서 문제를 찾아야 했음
- **A**: KODEX 200, TIGER S&P500 두 종목으로 보조지표 조합 테스트 → 6개 컬러 토큰 결함 식별
- **R**: 지원동기에 "보조지표 3개+ 조합 시 색상 의미 분리" 구체적 제안 포함

### 사례 B — VOC 70% 비교 요청 → 듀얼뷰 설계
- **S**: 채널톡 로그 분석 중 비교 관련 문의가 전체의 70% 이상 차지 발견
- **T**: 사용자가 우회하고 있는 진짜 니즈를 제품으로 해결해야 했음
- **A**: 캡처/새 창 우회 행동을 관찰하고, 인지심리학(단기기억 한계) 기반으로 듀얼뷰 설계
- **R**: 검색량 +26.43% 향상, 특허 등록

---

## 2. 최고를 추구

### 사례 A — 4층 토큰 → 단순 정리가 아닌 시스템적 해결
- **S**: 디자인 시스템 구축 시 단순 색상 정리로도 충분했으나, 근본적 구조 문제 인식
- **T**: "왜 같은 빨강이 다른 의미로 쓰이는가"를 원인부터 해결해야 했음
- **A**: 14만줄 코드 분석 → 재사용 패턴 4유형 발견 → 4층 계층화 + Do's/Don'ts 원칙 시스템 구축
- **R**: 빌드 -98%, 코드 -40%, "새 차트 추가 시에도 일관된 규칙 유지" 구조적 해결

### 사례 B — 특허 등록까지 추구
- **S**: 듀얼뷰 동기화 기능이 기술적으로 독창적이라는 피드백
- **T**: 회사 IP로 보호할 가치가 있다고 판단
- **A**: 변리사 미팅 → 기술 명세서 직접 작성 → 청구항 6개 정리 → 출원
- **R**: 특허 등록 완료, 회사 기술 자산으로 확보

## 3. 함께 성장

### 사례 A — FE 4명 + PD 2명 디자인시스템 크로스펑셔널 협업
- **S**: 디자인 시스템 도입 시 개발팀 전체의 동의와 참여가 필요
- **T**: 디자이너 혼자 만든 시스템이 아닌, 개발팀이 실제로 사용하는 시스템을 만들어야 했음
- **A**: JSON 토큰 파일 공유 체계, FE와 1:1 매핑 원칙 수립, PR 리뷰 참여, 주간 싱크 미팅
- **R**: 전체 팀이 동일 토큰 체계를 사용, 디자인-개발 간 소통 비용 대폭 감소

### 사례 B — PM/FE 리드와 사용자 데이터 기반 의사결정
- **S**: pathDual 프로젝트에서 3가지 비교 모드 중 우선순위 결정 필요
- **T**: 개인 직관이 아닌 데이터 기반으로 팀 합의 도출
- **A**: 사용 빈도 데이터 공유 → 동기화 모드 80% 확인 → 우선 개발 대상 합의
- **R**: 제한된 리소스로 가장 임팩트 큰 모드부터 출시, 팀 전체 납득

## 4. 투명하게 소통

### 사례 A — One Pager 문서화 습관
- **S**: 매 프로젝트 시작 시 One Pager(문제-목표-가설-범위)를 작성
- **T**: 이해관계자 전원이 같은 맥락에서 출발하도록 투명하게 공유
- **A**: Notion에 One Pager 작성 → 팀 리뷰 → 피드백 반영 후 킥오프
- **R**: "왜 이걸 하는지" 되묻는 상황 감소, 스코프 크리프 방지

### 사례 B — 개발 프로세스 도식화
- **S**: 디자인 시스템 도입 과정에서 "어디까지 진행됐는지" 혼란 발생
- **T**: 진행 상황을 누구나 한눈에 파악할 수 있는 구조 필요
- **A**: 전체 마이그레이션 프로세스를 단계별 플로우차트로 도식화 → 주간 업데이트
- **R**: 팀 전체가 현재 단계와 다음 단계를 명확히 인지

## 5. 빠르게 실행

### 사례 A — aiAgent 7단계 파이프라인 MVP 단계별 리드
- **S**: AI 에이전트 기능이 복잡하여 한 번에 출시하기 어려운 상황
- **T**: 불확실한 AI 기능을 빠르게 검증하면서도 품질을 유지해야 했음
- **A**: 7단계 파이프라인을 MVP 단위로 쪼개어 단계별 출시 → 각 단계에서 사용자 피드백 수집
- **R**: 554명 활성 사용자 확보, 99% 정확도 달성, 위험 최소화하며 빠른 출시

### 사례 B — 디자인 시스템 점진적 마이그레이션
- **S**: 14만줄 레거시를 한 번에 전환하면 서비스 중단 위험
- **T**: 서비스 운영을 유지하면서 점진적으로 전환해야 했음
- **A**: 컴포넌트 우선순위(사용 빈도 상위 23개) 선정 → 단계별 마이그레이션 → 각 단계 QA
- **R**: 서비스 중단 없이 전환 완료, 빌드 -98%, 코드 -40%
