# Portfolio Review: key.json 검수

이 프로젝트의 `portfolio_analysis.md` 전략 문서를 기준으로 `key.json`의 포트폴리오 텍스트 콘텐츠를 검수합니다.

## 실행 절차

1. `portfolio_analysis.md`를 읽어 검수 기준을 파악하세요.
2. `key.json`을 읽어 전체 텍스트 콘텐츠를 파악하세요.
3. 아래 **검수 체크리스트**를 각 프로젝트(myKeyword_project~aiAgent_project, dangen_project)별로 적용하세요.
4. 결과를 **리포트 형식**으로 출력하고, 각 이슈에 대해 **구체적인 수정안**을 제안하세요.

## 검수 체크리스트

### 1. 가독성 (핵심 요약)

- [ ] 각 텍스트가 1~2문장으로 압축되어 있는가?
- [ ] HEADLINE이 프로젝트의 핵심 성과를 즉각적으로 전달하는가?
- [ ] SUMMARY가 짧고 임팩트 있게 작성되었는가?
- [ ] 불필요하게 장황한 설명이 있는 텍스트는 없는가?

### 2. 논리 흐름 (문제 정의 -> 목표 -> 해결 -> 결과)

- [ ] PROBLEM → STRATEGY → SOL_A/B → IMPACT 순서로 논리가 끊기지 않고 이어지는가?
- [ ] PROBLEM에서 정의한 문제가 STRATEGY의 가설/원인과 직접 연결되는가?
- [ ] SOL_A/B가 STRATEGY에서 세운 가설의 해결책으로 자연스럽게 이어지는가?
- [ ] IMPACT의 성과가 처음 정의한 PROBLEM의 해결을 증명하는가?

### 3. 데이터 근거 (정량적 수치 및 정성적 인터뷰/리뷰)

- [ ] PROBLEM_QUANT에 구체적인 정량 데이터(수치, 비율 등)가 있는가?
- [ ] PROBLEM_QUAL에 실제 사용자 인용이나 정성적 근거가 있는가?
- [ ] IMPACT_OUTCOME에 디자인의 성공을 증명하는 데이터가 포함되어 있는가?
- [ ] 데이터가 누락된 프로젝트는 없는가? (null 값 체크)

### 4. 사용자 중심 (페르소나보다 문제점 도출)

- [ ] PROBLEM이 실제 사용자의 페인 포인트에서 출발하는가?
- [ ] 장황한 리서치 과정 나열 없이 핵심 문제가 명확히 드러나는가?
- [ ] "유저가 누구이고 그들의 문제가 무엇인지" 한눈에 파악 가능한가?

### 5. 뻔한 표현 탐지

다음과 같은 일반적/당연한 표현이 있으면 경고하세요:

- "직관적인 레이아웃", "직관적인 뷰", "직관적으로 알 수 있도록"
- "사용성을 쉽게 개선", "사용성을 개선하였고"
- "깔끔한 디자인", "깔끔한 레이아웃"
- "편리한 사용자 경험", "편리하게 사용할 수 있도록"
- "공감하는 디자이너"
- 기타 누구나 쓸 수 있는 미사여구나 모호한 표현

### 6. 기능 정의서화 탐지

다음과 같은 단순 기능 설명 패턴이 있으면 경고하세요:

- "~이 가능합니다", "~할 수 있습니다" 로만 끝나는 설명
- 디자인 의사결정의 근거 없이 기능만 나열하는 텍스트
- "왜 이 디자인을 선택했는가"가 빠진 솔루션 설명

## 리포트 출력 형식

각 프로젝트별로 다음 형식을 사용하세요:

```
## [프로젝트 ID] 검수 결과

### PASS 항목
- (통과한 체크리스트 항목 요약)

### ISSUE 항목
각 이슈에 대해:
- **[카테고리]** 이슈 설명
  - 현재: `해당 key.json 키` = "현재 텍스트"
  - 문제: 구체적인 문제점 설명
  - 수정안: "제안하는 개선된 텍스트"

### 종합 점수
- 가독성: O/X
- 논리 흐름: O/X
- 데이터 근거: O/X
- 사용자 중심: O/X
- 뻔한 표현 없음: O/X
- 기능 정의서화 없음: O/X
```

## 주의사항

- 수정안은 반드시 한국어로 작성하세요.
- key.json의 키 이름은 변경하지 마세요. 값(텍스트)만 수정 제안하세요.
- 수정안 제안 시, portfolio_analysis.md의 원칙을 근거로 제시하세요.
- 사용자가 수정을 승인하면 key.json에 직접 반영하세요.
